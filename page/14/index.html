flex-grow 和 flex-shrink

flex-grow 属性的作用：

父元素在主轴方向还有剩余空间时，子元素们如何分配这些剩余空间。

其值为一个权重（也称扩张因子），默认为 0（纯数值，无单位），剩余空间将会按照这个权重来分配。

比如剩余空间为 x，三个元素的 flex-grow 分别为 a，b，c，求和 sum = a + b + c。那么三个元素权重是 a/sum, b/sum, c/sum，就是按照这个比例分配剩余空间的。

举个例子：

父元素宽度 500px，三个子元素的 width 分别为 100px，150px，100px，于是剩余空间为 150px。

如果三个元素的 flex-grow 分别是 1，2，3，那么 sum 为 6，则三个元素被分到的空间是：

  • 150 * 1 / 6 = 25px
  • 150 * 2 / 6 = 50px
  • 150 * 3 / 6 = 75px

三个元素最终的宽度分别：

  • 100px + 25px = 125px
  • 150px + 50px = 200px
  • 100px + 75px = 175px

双指针在算法题中的应用

双指针是一种巧妙的解题思路，我们通过几道题来体验一下：

求链表的中间结点

题目描述：给定一个带有头结点 head 的非空单链表，返回链表的中间结点。如果有两个中间结点，则返回第二个中间结点。

此题并不难，假设链表中有 N 个节点，就是需要找到索引为 Math.floor(N/2) 的节点即可，例如：

  • [1,2,3,4,5] 返回 3，即第 2 个节点
  • [1,2,3,4,5,6] 返回 4，即第 3 个节点

其实更好的解法是使用双指针，更准确叫「快慢指针」，fast 和 slow 指针开始都指向 head，然后：

  • fast 移动两步
  • slow 移动一步

那么 fast 指针和 slow 指针所在索引的对应关系为：

  • fast=0, slow=0，示例 [0]
  • fast=2, slow=1，示例 [0, 1, 2]
  • fast=4, slow=2，示例 [0, 1, 2, 3, 4]
  • fast=6, slow=3，示例 [0, 1, 2, 3, 4, 5, 6]
  • ….

盛水最多的容器

题目描述：给你 n 个非负整数 [a1，a2，...，an]，每个数代表坐标中的一个点 (i, ai) 。在坐标内画 n 条垂直线，垂直线 i 的两个端点分别为 (i, ai) 和 (i, 0)。找出其中的两条线，使得它们与 x 轴共同构成的容器可以容纳最多的水。例如：

  • 输入：[1,8,6,2,5,4,8,3,7]
  • 输出：49

也就是说，要从数组中找到两个值 ai 和 aj，使得 Min(ai, aj)*(j-i) 最大。

这道题当然也是可以用穷举法来做的，算出所有可能的值，但是时间复杂度为 O(n^2)，更好的解决方案就是双指针，更准确的说是「头尾指针」，一开始的时候，让头指针指向数组开始元素，尾指针指向数组最后元素，计算其容量。接下来只可能出现两种移动方式：

  • 头指针右移一步
  • 尾指针左移一步

微服务学习笔记

把传统业务拆分成微服务的时候，要注意下面的要点：

  1. 区分边界服务和内部服务。边界服务是会接受公网流量的服务，比如处理HTTP请求的服务；反之就是内部服务。边界服务和内部服务在安全方面的考虑是不一样的，例如边界服务需要做针对User的鉴权等等。
  2. 区分基础和业务服务。业务服务追求变化，feature，基础服务追求稳定，性能。
  3. 一定要考虑调用失败的场景。微服务的提供方，一定要考虑调用可能会有bug，比如出现死循环。
  4. 没有把握的做到合理拆分的时候可以选择先不拆，等到发现瓶颈了自然就知道怎么拆了。

从通信类型的角度看，大概有三种类型：

  1. 同步调用
  2. 异步调用
  3. 广播

在微服务的设计之初要想清楚调用关系以及调用方式，哪些需要同步，哪些可以异步，哪些需要广播，最好团队内部要有统一的认识。然后就是要确定好调用协议了，例如常见的选择有：

  • 同步调用：HTTP REST, gRPC, thrift, etc.
  • 异步调用：Sidekiq, Celery, etc. 对应的backend有 Redis, 各类 AMQP 实现, Kafka 等等
  • 广播：各类 AMQP 实现, Kafka，etc.

微服务除了是个技术活，更是个管理活。一定得有和微服务相融的团队组织方式和工作流程。首先就是要成立架构组。架构组要负责微服务的基础设施的建设，例如 CI/CD 系统， Kubernetes 集群，Service Mesh 集群，监控报警系统，日志手机系统，基础的工具和库的构建和维护。在我们的实践中，每个微服务团队由2-3人组成，有一个master，负责 DevOps 流程的建设，权限的分配等等一系列工作。例如我们的DevOps是基于 GitLab CI 和 kubernetes 的，所以 master 要负责 .gitlab-ci.yml ，以及所有自己服务相关的 k8s 的 deployment, job, service 等等的编写维护。

【备注】此笔记中的观点来自网络收集，用于记录微服务最佳实践，学习共勉。
