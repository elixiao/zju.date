微信扫码支付

信扫码支付，官方提供了两种模式：

  1. 商户根据微信支付规则链接生成二维码，带固定参数productid（订单号）。用户扫码后，微信支付系统将productid和用户唯一标识（openid）回调商户后台系统，商户后台系统根据productid生成支付交易，最后微信支付系统发起用户支付流程。
  2. 商户后台系统调用微信支付预付交易接口，将返回的链接生成二维码，用户扫码后输入密码完成支付交易。

模式2比较简单，逐一分析。

模式2

首先调用统一下单接口地址：https://api.mch.weixin.qq.com/pay/unifiedorder，这个地址需要提供以下参数，放在body里面通过POST请求调用：

feathersjs hook

feathersjs hook 对象里面究竟都有什么？

  1. hook.app

    只读属性，是 feathersjs 对象，可用于获取其他 service，例如：hook.app.service('name')

  1. hook.service

    只读属性，当前 hook 执行所在的 service

  2. hook.path

    只读属性，表示当前 service 的路径

  3. hook.method

    只读属性，是 find, get, create, update, patch, remove 中的一个

  4. hook.type

    只读属性，是 before, after, error 中的一个

  5. hook.params

    可写属性，包含客户端传过来的参数，例如 params.query

  6. hook.id

    可写属性， get, remove, update 和 patch 方法才有

  7. hook.data

    可写属性，create, update 和 patch 方法才有

  8. hook.error

    可写属性，只有当 hook.type = error 时才有

  9. hook.result

    可写属性，只有当 hook.type = after 时才有，但是可以在 before 或 error hooks 中设置它。

ElasticSearch学习笔记

ElasticSearch概述

与传统数据库的类比：

关系数据库      ⇒ 数据库(database) ⇒ 表(table)    ⇒ 行(row)         ⇒ 列(column)
Elasticsearch  ⇒ 索引(index)      ⇒ 类型(type)   ⇒ 文档(document)  ⇒ 字段(field)

一个 Elasticsearch 集群可以包含多个索引，每个索引可以包含1个类型，这个类型中包含多个文档，然后每个文档中又包含了很多的字段。但是在 6.x 的版本中 type 被废弃了，在后期的版本中被完全移除，因此可以把 ES 想象成一个数据库只能容纳一个表。废弃的原因文档里面说得也比较清楚，在传统数据库中，表跟表之前是相互独立的，可以拥有不同的字段、不同的索引，但是在 ES 中，同一个索引下不同类型里名称相同的字段会被 Lucene 视为同一个字段，这就会产生矛盾，例如两个类型里面都有 deleted 字段，想在其中一个类型定义它是日期格式，另一个类型中定义它是布尔值，ES 做不到这一点，但数据库是可以做到的。
